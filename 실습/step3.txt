현재 디렉토리의 AORR.md를 읽고, 해당 개발 업무를 Verifier 중심의 Self-Correcting TDD 루프로 설계해줘.

다음 내용을 포함해:

1. 테스트 순서
   - 기존 테스트 또는 재현 테스트
   - Unit Test
   - Lint
   - Type Check
   - Build
   - 필요한 경우 화면 또는 기능 확인

2. 테스트 명령어
   - 프로젝트에서 실제 실행 가능한 명령어를 확인
   - 확인할 수 없는 명령어는 임의로 만들지 말고 [사람 확인 필요]로 표시

3. 실패 로그 관찰 방법
   - exit code
   - 실패한 테스트 이름
   - 핵심 에러 메시지
   - 관련 파일과 라인
   - 동일 오류 여부를 판단하기 위한 오류 fingerprint

4. 실패 원인 분류
   - CODE
   - TEST
   - SPEC
   - ENVIRONMENT
   - PERMISSION
   - UNKNOWN

5. 최소 수정 원칙
   - 한 번의 Retry에서는 하나의 원인만 수정
   - 관련 파일만 변경
   - 테스트를 통과시키기 위한 우회, 테스트 삭제, 검증 완화 금지
   - 대규모 리팩토링 금지

6. Retry 정책
   - 최대 3회
   - 동일 오류 fingerprint가 2회 반복되면 즉시 정지
   - Retry마다 가설, 변경 파일, 테스트 결과를 기록

7. Stop 및 HITL 조건
   - 전체 Verifier 통과 시 PASSED
   - Retry 한계 도달 시 HITL_REQUIRED
   - 스펙, 권한, 배포, 외부 서비스 판단이 필요하면 HITL_REQUIRED
   - 환경 문제로 테스트를 실행할 수 없으면 BLOCKED

가능하면 Claude Code CLI를 독립적인 Verifier로 사용하는 방법을 포함해줘.
Claude Code CLI에서 Sonnet 5 사용 가능 여부를 먼저 확인하고, 가능하면 Sonnet 5를 사용해.
사용할 수 없다면 현재 사용 가능한 Sonnet 모델을 사용하고 실제 모델명을 기록하도록 해줘.

이 단계에서는 루프를 실행하거나 코드를 수정하지 말고 설계만 수행해줘.
결과를 AORR.md의 기존 내용 아래에 'Self-Correcting TDD Loop' 섹션으로 추가해줘.
