Step 1에서 분석한 개발 업무와 clone된 GitHub 저장소를 기준으로, 개인 프로페셔널 웹사이트 개발 업무를 실행 가능한 AORR 상태 머신으로 설계해줘.

최종 결과물은 GitHub Pages에서 실행 가능한 정적 웹사이트이며, 최소한 다음 파일이 루트 디렉토리에 존재해야 해:

- index.html
- styles.css
- script.js
- 게임 구현에 필요한 JavaScript 파일 또는 script.js 내부 게임 코드

별도의 백엔드 서버 없이 HTML, CSS, JavaScript만으로 동작해야 해.

다음 항목을 포함해:

1. Target
   - 프로페셔널 웹사이트 개발 목표
   - GitHub Pages 배포 목표
   - 입력 자료
   - 필수 페이지와 섹션
   - Games 탭 및 지렁이 게임 요구사항
   - 데스크톱 및 모바일 완료 기준

2. Act
   - 한 번의 개발 루프에서 수행할 최소 작업
   - 수정 가능한 파일 범위
   - 생성할 수 있는 파일
   - 실행 가능한 로컬 검증 명령어

3. Observe
   - 파일 생성 여부
   - HTML, CSS, JavaScript 오류
   - 로컬 웹서버 응답
   - 브라우저 콘솔 오류
   - 데스크톱 및 모바일 화면
   - 키보드 및 터치 게임 조작
   - GitHub Pages 호환성

4. Reason
   실패 원인을 다음 중 하나로 분류하는 기준:
   - HTML_STRUCTURE
   - CSS_RESPONSIVE
   - JAVASCRIPT
   - GAME_LOGIC
   - GAME_CONTROL
   - CONTENT
   - TEST
   - ENVIRONMENT
   - GITHUB_PERMISSION
   - DEPLOYMENT
   - UNKNOWN

5. Repeat
   - 한 번에 하나의 실패 원인만 수정
   - 관련된 최소 파일만 변경
   - 수정 후 동일한 Verifier 재실행
   - 기존에 통과한 기능에 대한 회귀 테스트 실행

6. Stop
   - 전체 테스트가 통과한 경우
   - 최대 Retry에 도달한 경우
   - 동일한 오류 fingerprint가 2회 반복된 경우
   - 개인정보나 콘텐츠 확인이 필요한 경우
   - GitHub 인증 또는 배포 권한 문제가 발생한 경우

7. Human-in-the-loop
   - 이름, 소개, 경력, 프로젝트 등 개인 콘텐츠가 불명확한 경우
   - 기존 콘텐츠 삭제가 필요한 경우
   - 외부 분석 도구나 외부 서비스를 추가해야 하는 경우
   - GitHub 저장소 설정을 변경해야 하는 경우
   - 요구사항이 충돌하는 경우

전체 개발 업무를 다음과 같은 작은 루프로 나눠줘:

- 저장소 및 기존 파일 확인
- 정적 사이트 기본 구조
- 프로페셔널 콘텐츠 영역
- 반응형 내비게이션
- Games 탭
- 지렁이 게임 핵심 로직
- 키보드 조작
- 모바일 터치 조작
- 게임 UI 및 점수
- 접근성과 반응형 검증
- GitHub Pages 호환성 검증
- 배포

각 단계의 입력, Act, Observe, 출력, 테스트 기준, 다음 상태를 표로 정리해줘.

상태는 다음 중 하나를 사용해:

READY
ACTING
VERIFYING
RETRYING
PASSED
DEPLOY_READY
DEPLOYING
DEPLOYED
BLOCKED
HITL_REQUIRED

Step 1에 [게임 추가 기능:]이 작성되어 있다면 해당 요구사항도 게임 루프에 반영해줘.

확실하지 않은 내용은 추측하지 말고 [사람 확인 필요]로 표시해줘.

아직 코드를 수정하거나 테스트하거나 배포하지 말고 설계만 수행해줘.

결과를 현재 프로젝트 루트의 AORR.md에 저장해줘.
