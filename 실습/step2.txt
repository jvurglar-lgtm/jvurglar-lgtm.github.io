현재 개발 티켓과 관련 내용을 확인하고, 이 개발 업무를 실행 가능한 AORR 상태 머신으로 설계해줘.

다음 항목을 포함해:

1. Target
   - 개발 목표
   - 입력 자료
   - 완료 기준

2. Act
   - 한 번의 루프에서 수행할 작업
   - 수정 가능한 파일 범위
   - 실행 가능한 명령어

3. Observe
   - 확인할 테스트 결과, exit code, 빌드 로그, 화면 동작
   - 성공과 실패를 판단할 객관적인 기준

4. Reason
   - 실패 원인을 코드, 테스트, 요구사항, 환경, 권한 문제로 분류하는 기준

5. Repeat
   - 한 번에 하나의 원인만 수정
   - 최소한의 파일과 코드만 변경
   - 수정 후 동일한 Verifier를 다시 실행

6. Stop
   - 모든 테스트가 통과한 경우
   - 최대 Retry 횟수에 도달한 경우
   - 동일한 오류가 2회 반복된 경우
   - 요구사항 또는 환경이 불명확한 경우

7. Human-in-the-loop
   - 배포, API 스펙, 인증·권한, 외부 서비스, 대규모 리팩토링처럼 사람 승인이 필요한 조건

각 단계의 입력, 행동, 출력, 다음 상태를 표로 정리해줘.
상태는 READY, ACTING, VERIFYING, RETRYING, PASSED, BLOCKED, HITL_REQUIRED 중 하나로 표현해줘.

확실하지 않은 내용은 추측하지 말고 [사람 확인 필요]로 표시해줘.
아직 코드를 수정하거나 테스트를 실행하지 말고 설계만 수행해줘.

결과를 현재 디렉토리의 AORR.md에 저장해줘.
