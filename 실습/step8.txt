Step 7에서 배포된 GitHub Pages 웹사이트를 사용자가 확인했어.

다음 사용자 수정 요청을 새로운 변경 업무로 정리해줘.

사용자 수정 요청:
"""
[배포된 웹사이트를 보고 수정하고 싶은 내용을 여기에 입력]
"""

추가 자료:
"""
[필요하면 문제가 발생한 화면, 모바일 기기, 브라우저, 재현 방법 등을 입력]
"""

현재 프로젝트의 다음 자료를 읽어:

- 배포된 웹사이트
- 현재 소스 코드
- AORR.md
- MEMORY.md
- 마지막 배포 commit
- 기존 테스트 결과

다음 작업을 수행해:

1. 사용자 요청 원문 보존
   - 사용자의 표현을 임의로 바꾸거나 누락하지 않음

2. 수정 요청 분류
   - BUG
   - UI_UX
   - RESPONSIVE
   - CONTENT
   - GAME_LOGIC
   - GAME_CONTROL
   - ACCESSIBILITY
   - NEW_FEATURE
   - SPEC_CHANGE
   - DEPLOYMENT
   - UNKNOWN

3. 현재 동작과 기대 동작 비교
   - 현재 배포된 사이트의 동작
   - 사용자가 기대하는 동작
   - 차이점
   - 재현 방법

4. 수정 범위 결정
   - 변경 가능성이 있는 파일
   - 변경해야 하는 기능
   - 변경하지 말아야 하는 기능
   - 기존 정상 기능에 대한 회귀 위험

5. 완료 기준 작성
   - 사용자의 요청을 객관적으로 확인할 수 있는 기준
   - 데스크톱 검증 기준
   - 모바일 검증 기준
   - 게임 관련 검증 기준
   - GitHub Pages 배포 후 검증 기준

6. 새로운 테스트 작성 계획
   - 수정 전 실패를 재현하는 테스트
   - 수정 후 통과해야 하는 테스트
   - 기존 기능 회귀 테스트

7. 위험도
   - LOW
   - MEDIUM
   - HIGH

8. 루프 재진입 판단
   - 요구사항이 명확하면 CHANGE_READY
   - 요구사항이 불명확하면 HITL_REQUIRED
   - 기존 요구사항과 충돌하면 HITL_REQUIRED

변경 요청을 CHANGE_REQUEST.md에 저장해.

CHANGE_REQUEST.md에는 다음 내용을 포함해:

- Change Request ID
- 사용자 요청 원문
- 요청 분류
- 현재 동작
- 기대 동작
- 재현 방법
- 수정 범위
- 변경 금지 범위
- 완료 기준
- 새 테스트
- 회귀 테스트
- 위험도
- 예상 수정 파일
- 배포 필요 여부
- 사람 확인 필요 항목

AORR.md 아래에는 이번 변경 요청에 대한 새로운 AORR Change Loop를 추가해.

MEMORY.md에는 다음 내용을 업데이트해:

- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 새로운 Change Request ID
- 사용자 요청 요약
- 현재 상태
- 새 완료 기준
- 다음 Step 9에서 실행할 첫 번째 작업

기존 실행 기록과 기존 완료 기준은 삭제하거나 덮어쓰지 마.

이 단계에서는 다음 작업을 하지 마:

- 코드 수정
- commit
- push
- GitHub Pages 재배포
- 기존 테스트 결과 삭제

불명확한 내용은 임의로 결정하지 말고 [사람 확인 필요]로 표시해줘.

최종적으로 다음을 보고해:

- Change Request ID
- 요청 분류
- 수정 범위
- 완료 기준
- 위험도
- 예상 수정 파일
- Step 9 재루프 실행 가능 여부
- [사람 확인 필요] 항목
