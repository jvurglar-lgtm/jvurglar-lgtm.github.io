배포된 웹사이트를 사용자가 검수한 뒤 전달한 수정 요청을 새로운 변경 티켓으로 정리해줘.

사용자 수정 요청:
"""
[여기에 사용자가 웹사이트를 보고 전달한 피드백 입력]
"""

현재 디렉토리의 AORR.md와 MEMORY.md를 읽고 다음 작업을 수행해:

1. 기존 완료 상태 확인
   - 기존 목표와 통과한 테스트
   - 현재 배포 상태
   - 마지막 정상 버전
   - 기존 실행 기록은 수정하거나 삭제하지 않음

2. 사용자 피드백 분류
   - BUG: 기존 요구사항과 다르게 동작
   - UI_UX: 디자인, 배치, 반응형, 사용성 변경
   - NEW_FEATURE: 새로운 기능 요청
   - SPEC_CHANGE: 기존 요구사항 변경
   - UNKNOWN: 요구가 불명확함

3. 변경 티켓 작성
   다음 내용을 포함해 CHANGE_REQUEST.md에 저장해:
   - 변경 요청 ID
   - 사용자 요청 원문
   - 현재 동작
   - 기대 동작
   - 수정 범위
   - 수정 금지 범위
   - 완료 기준
   - 검증 방법
   - 기존 기능 회귀 테스트 기준
   - 우선순위
   - 사람 확인 필요 항목

4. 영향도 분석
   - 변경될 가능성이 있는 파일
   - 기존 테스트에 미치는 영향
   - 새로 추가해야 할 테스트
   - 배포가 필요한지 여부
   - 위험도: LOW, MEDIUM, HIGH

5. 루프 재진입 판단
   - 요구사항이 충분히 명확하면 상태를 CHANGE_REQUESTED로 설정
   - 요구사항이 불명확하면 HITL_REQUIRED로 설정
   - API 스펙, 인증·권한, 외부 서비스 변경이 포함되면 사람 승인을 요청

6. MEMORY.md 업데이트
   기존 실행 기록은 유지하고 다음 내용을 추가해:
   - 새로운 변경 요청 ID
   - 사용자 피드백 요약
   - 현재 상태
   - 새 완료 기준
   - 다음 루프에서 수행할 첫 작업

이 단계에서는 코드를 수정하거나 배포하지 마.
불명확한 내용은 임의로 결정하지 말고 [사람 확인 필요]로 표시해줘.
