현재 프로젝트의 CHANGE_REQUEST.md, AORR.md, MEMORY.md, 현재 코드와 마지막 정상 배포 commit을 읽고 사용자 수정 요청에 대한 새로운 Self-Correcting TDD 재루프를 실행해줘.

이번 루프의 목표는 Step 8에서 확정한 사용자 수정사항만 안전하게 반영하고, 기존 기능을 유지한 상태로 GitHub Pages에 재배포하는 것이야.

기존 완료 기록과 실행 기록은 삭제하거나 덮어쓰지 마.

먼저 다음을 확인해:

1. CHANGE_REQUEST.md의 사용자 요청 원문
2. 현재 동작과 기대 동작
3. 완료 기준
4. 수정 허용 범위
5. 변경 금지 범위
6. 마지막 정상 배포 commit
7. 기존 회귀 테스트
8. 현재 Retry 횟수

변경 작업은 다음 순서로 수행해:

1. 수정 전 문제 재현
2. 실패 테스트 또는 재현 결과 기록
3. 실패 원인 분류
4. 가장 작은 수정 하나 수행
5. 동일한 테스트 재실행
6. 새로운 변경 요청 완료 기준 검증
7. 기존 프로페셔널 웹사이트 회귀 테스트
8. 기존 지렁이 게임 회귀 테스트
9. 모바일 및 데스크톱 검증
10. GitHub Pages 호환성 검증
11. MEMORY.md 실행 기록 업데이트

실패 원인은 다음 중 하나로 분류해:

HTML_STRUCTURE
CSS_RESPONSIVE
JAVASCRIPT
GAME_LOGIC
GAME_CONTROL
CONTENT
ACCESSIBILITY
TEST
ENVIRONMENT
GITHUB_PERMISSION
DEPLOYMENT
UNKNOWN

Retry 정책:

- 하나의 오류당 최대 3회
- 동일 오류 fingerprint가 2회 반복되면 중지
- 한 번의 Retry에서는 하나의 원인만 수정
- CHANGE_REQUEST.md에 정의된 범위 밖의 파일은 수정하지 않음
- 테스트 삭제 또는 완료 기준 완화 금지
- 기존 기능을 제거해 문제를 해결하지 않음
- 불필요한 전체 사이트 재작성 금지

반드시 다음을 검증해:

- 사용자 수정 요청 완료 기준
- 기존 내비게이션
- 기존 프로필 콘텐츠
- 기존 모바일 반응형
- Games 탭
- 지렁이 게임 시작, 정지, 재시작
- 키보드 조작
- 모바일 터치 조작
- 점수와 충돌 처리
- 브라우저 콘솔 오류
- 깨진 링크
- GitHub Pages 상대 경로
- 모바일, 태블릿, 데스크톱 화면

가능하면 Claude Code CLI를 독립 Verifier로 사용해.
Sonnet 5가 실제 사용 가능하면 사용하고, 그렇지 않으면 사용 가능한 Sonnet 모델을 사용해.
실제 사용 모델을 MEMORY.md에 기록해.

모든 변경 테스트와 회귀 테스트가 통과하면 재배포를 수행해.

재배포 절차:

1. 변경 파일과 Git diff 검토
2. 변경 범위가 CHANGE_REQUEST.md와 일치하는지 확인
3. github_token.txt 및 비밀정보 파일이 Git에 포함되지 않았는지 확인
4. 코드와 diff에 토큰 문자열이 없는지 검사
5. Change Request ID가 포함된 commit 메시지 작성
6. 현재 디렉토리의 github_token.txt에 있는 token="" 값을 안전하게 사용
7. 토큰을 출력하거나 코드, remote URL, Git 기록에 저장하지 않음
8. GitHub 저장소에 push
9. GitHub Pages 재배포 완료 확인
10. 배포된 사이트에서 사용자 수정사항 확인
11. 전체 회귀 테스트 재실행
12. CHANGE_REQUEST.md 상태를 DEPLOYED로 업데이트
13. MEMORY.md에 새로운 정상 배포 commit과 URL 기록

토큰 인증을 위한 임시 파일이나 환경 변수를 만들었다면 작업 후 즉시 제거해.
