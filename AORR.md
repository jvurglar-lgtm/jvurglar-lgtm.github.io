# AORR 상태 머신

## 1. Target과 완료 기준

### Target
- `https://jvurglar-lgtm.github.io`에서 동작하는 정적 개인 프로페셔널 웹사이트를 완성한다.
- 웹사이트는 `HTML`, `CSS`, `JavaScript`만 사용한다.
- 데스크톱, 태블릿, 모바일에서 반응형으로 보인다.
- 상단 `Games` 메뉴에서 지렁이 게임으로 진입할 수 있다.
- 지렁이 게임은 키보드와 모바일 터치로 조작 가능하다.
- GitHub Pages에서 정상 배포 가능해야 한다.

### 완료 기준
- 홈, 소개, 프로젝트/경험, 연락처 등 핵심 개인 웹사이트 정보가 보인다.
- `Games` 메뉴가 상단 네비게이션에 존재하고 접근 가능하다.
- 지렁이 게임이 시작, 이동, 충돌, 점수, 재시작의 기본 흐름을 제공한다.
- 모바일에서 레이아웃이 깨지지 않고 게임 조작이 가능하다.
- 로컬 검증과 GitHub Pages 호환성 검증이 통과한다.

## 2. Act: Codex가 수행할 최소 수정

Codex는 한 번의 Retry마다 아래 원칙을 따른다.

- 한 번에 하나의 실패 원인만 고친다.
- 관련 파일만 최소 범위로 수정한다.
- 구조를 유지하고, 대규모 재작성은 피한다.
- 테스트를 완화하거나 삭제하지 않는다.

최소 수정 예시:
- HTML 구조 문제: `index.html`의 잘못된 태그, 링크, 섹션 구조만 수정
- CSS 문제: 반응형 브레이크포인트, 간격, 가시성, 레이아웃만 수정
- JavaScript 문제: 게임 이동, 충돌, 입력, 상태 전환의 로직만 수정
- 콘텐츠 문제: 프로필 문구, 메뉴명, 게임 안내 문구만 수정
- GitHub 문제: 배포 경로, 상대경로, 정적 자산 참조만 수정

## 3. Observe: Claude가 실행할 테스트와 수집할 결과

Claude Code CLI Sonnet이 기본 Verifier다. 가능한 경우 Codex는 Claude가 수행한 테스트를 중복 실행하지 않는다.

### Observe 범위
- 파일 존재와 상대 경로 확인
- HTML 구조와 내부 링크 확인
- CSS 반응형 확인
- JavaScript 오류 확인
- 지렁이 게임 기능과 입력 확인
- 로컬 HTTP 응답 확인
- 뷰포트 확인: 375px, 768px, 1440px
- GitHub Pages 호환성 확인

### 수집할 결과
- 실행 주체와 모델명
- 실행 명령
- exit code
- 실패한 항목
- 관련 파일과 라인
- fingerprint
- 최종 상태

### Claude 사용 불가 시
- `CODEX_FALLBACK`으로 기록한다.
- 이 경우 Codex가 변경과 테스트를 모두 수행한다.
- 현재 환경에서는 `claude_code.exe`를 사용한다.
- `ANTHROPIC_API_KEY`가 우선되면 401이 발생하므로, Claude 실행 시 해당 변수를 제거해야 한다.

## 4. Reason: 실패 원인 분류

실패는 아래 중 하나로 분류한다.

- `HTML`: 잘못된 구조, 누락된 태그, 링크 문제
- `CSS`: 레이아웃 깨짐, 반응형 실패, 가시성 문제
- `JAVASCRIPT`: 런타임 오류, 입력 처리 실패, 상태 꼬임
- `GAME`: 지렁이 이동, 성장, 충돌, 점수, 재시작 문제
- `CONTENT`: 문구, 섹션, 링크 텍스트, 프로필 정보 문제
- `TEST`: 테스트 자체가 불안정하거나 잘못됨
- `ENVIRONMENT`: 로컬 실행 환경, 도구, 의존성, 파일 접근 문제
- `GITHUB`: 저장소 구조, Pages 경로, 토큰/권한 문제
- `DEPLOYMENT`: 배포 후 페이지 미노출, 404, 경로 오류
- `UNKNOWN`: 위 분류로 설명되지 않는 경우

## 5. Repeat: Codex 최소 수정 → Claude 동일 테스트 재실행

반복 규칙:

1. Claude가 실패를 보고한다.
2. Codex는 원인 하나에 해당하는 최소 파일만 수정한다.
3. Claude가 동일 테스트를 다시 실행한다.
4. 결과가 개선되지 않으면 같은 원인 안에서만 추가 최소 수정한다.
5. 동일 fingerprint가 반복되면 중지하고 HITL로 올린다.

제약:
- 한 Retry에서는 원인 하나만 다룬다.
- 테스트 삭제, 완화, 우회는 금지한다.
- 관련 없는 리팩터링은 하지 않는다.

## 6. Stop과 HITL 조건

다음 상황에서는 중지하거나 사람 확인을 요청한다.

- 동일 원인으로 3회 이상 실패
- 동일 fingerprint가 2회 이상 반복
- 프로필 내용, 이름, 링크, 공개 범위가 불명확함
- 게임 규칙, 적 추가 여부, 조작 방식이 불명확함
- GitHub Pages 또는 토큰 권한 문제로 진행 불가
- 배포 승인 전 단계에서 배포가 필요함

상태:
- `HITL_REQUIRED`: 사람 확인이 필요함
- `BLOCKED`: 기술적으로 더 진행할 수 없음
- `DEPLOY_APPROVAL_REQUIRED`: 배포 전 승인 필요

## 7. 상태 정의

- `READY`: 다음 루프를 시작할 준비 상태
- `ACTING`: Codex가 최소 수정 중
- `VERIFYING`: Claude가 테스트 중
- `RETRYING`: 실패 후 같은 원인으로 최소 수정 반복 중
- `PASSED`: 현재 루프의 검증이 통과됨
- `DEPLOY_APPROVAL_REQUIRED`: 배포 직전 승인 필요
- `DEPLOYED`: GitHub Pages 배포 완료
- `BLOCKED`: 같은 원인으로 더 진행 불가
- `HITL_REQUIRED`: 사람 입력이 필요한 상태

## 8. Self-Correcting TDD Loop

### 기본 원칙
- Verifier 우선: 가능하면 Claude가 변경 전과 변경 후 테스트를 모두 수행한다.
- Worker 최소 수정: Codex는 실패 원인 하나만 고친다.
- 동일 검증 반복: 같은 테스트 세트를 변경 전후에 유지한다.
- 중복 실행 방지: Claude가 수행한 테스트를 Codex가 다시 돌리지 않는다.

### 순서
1. Claude가 변경 전 테스트를 실행한다.
2. Claude가 실패 항목, 핵심 오류, 관련 파일, fingerprint를 보고한다.
3. Codex가 원인 하나에 필요한 최소 코드만 수정한다.
4. Claude가 동일 테스트를 재실행한다.
5. 실패가 남으면 Claude가 새 결과와 fingerprint를 보고한다.
6. Codex가 다시 최소 수정하고 Claude에 재검증을 요청한다.
7. Claude의 전체 테스트가 통과해야 `PASSED`가 된다.

### 기록 규칙
- 실패 기록에는 실행 주체와 모델, 명령, exit code, 핵심 오류, 관련 파일·라인, fingerprint, 최종 상태를 남긴다.
- Claude CLI를 사용할 수 없으면 `CODEX_FALLBACK`과 이유를 남긴다.
- 모델명이 확인되지 않으면 `[사람 확인 필요]`로 둔다.

### 현재 환경 상태
- Claude CLI 실행: 성공
- 실행 경로: `claude_code.exe`
- 실행 조건: `ANTHROPIC_API_KEY` 제거 필요
- 실행 모드: `CLAUDE_VERIFIER`
- Claude 모델: `claude-sonnet-4-6`
- 다음 조치: Claude를 Verifier로 사용하고 Codex는 최소 수정만 수행

## 10. 실행 원칙

- Codex는 문제를 작게 쪼갠 뒤 최소 수정만 한다.
- Claude는 같은 기준으로 검증한다.
- 검증이 통과하지 않으면 다음 상태로 가지 않는다.
- 사람이 정해야 하는 항목은 `[사람 확인 필요]`로 둔다.
